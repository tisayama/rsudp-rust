# Feature Specification: アラート音声の順序再生（キューイング）

**Feature Branch**: `044-audio-queue-playback`
**Created**: 2026-03-08
**Status**: Draft
**Input**: User description: "アラート音声の順序再生(キューイング): 現在、ALARM/RESETのMP3再生はtokio::task::spawn_blockingで独立スレッドを起動するため、ALARM音の再生中にRESET音が発火すると両方が同時に鳴る。音声再生をキューイング方式に変更し、先行する音声の再生完了後に次の音声を順番に再生する。既存の「再生ごとにALSAストリームを新規作成する」設計（Raspberry Piでの長時間稼働対策）は維持する。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 音声の順序再生 (Priority: P1)

地震監視システムの運用者として、ALARM音の再生中にRESETイベントが発生した場合、ALARM音が再生完了した後にRESET音が順番に再生されることを求める。これにより、各アラート音が途切れず明瞭に聞き取れる。

**Why this priority**: 現在の並行再生では2つの音声が重なり、どちらのイベントの音声かを聴覚で判別できない。順序再生は本機能の核心である。

**Independent Test**: ALARM音(例: 3秒)の再生中にRESETイベントを発火させ、ALARM音が完了した後にRESET音が開始されることを確認する。

**Acceptance Scenarios**:

1. **Given** ALARM音が再生中である, **When** RESETイベントが発生する, **Then** RESET音はALARM音の再生完了後に開始される
2. **Given** 再生キューが空である, **When** ALARMイベントが発生する, **Then** ALARM音が遅延なく即座に再生される
3. **Given** ALARM音もRESET音も再生されていない, **When** RESETイベントが発生する, **Then** RESET音が遅延なく即座に再生される

---

### User Story 2 - 長時間稼働での安定性維持 (Priority: P1)

Raspberry Pi上で24時間365日稼働するシステムの運用者として、音声キューイング導入後もALSAオーディオストリームの安定性が維持されることを求める。既存の「再生ごとにストリームを新規作成する」設計が保持され、長時間アイドル後のオーディオエラーが発生しないこと。

**Why this priority**: Raspberry PiでのALSAバッファエラー（POLLERR）回避は稼働安定性に直結するため、P1とする。

**Independent Test**: 再生完了後にオーディオストリームが解放されていることを確認する（永続的なストリーム保持がないこと）。

**Acceptance Scenarios**:

1. **Given** キュー内の音声が再生完了した, **When** 次のキュー項目がない, **Then** オーディオストリームは解放され、リソースを消費しない
2. **Given** 最後の音声再生から1時間以上経過した, **When** 新しいアラートイベントが発生する, **Then** 音声が正常に再生される（ALSAエラーなし）

---

### Edge Cases

- 短時間に3件以上のアラートイベントが連続発生した場合、すべての音声がキュー順に再生される
- 指定された音声ファイルが存在しない場合、そのキュー項目はスキップされ、次の項目が再生される
- 音声ファイルのパスが空文字の場合、キューに追加されない
- システム起動直後（まだ一度も再生していない状態）で正常にキューが機能する

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: システムは音声再生リクエストをキューに蓄積し、先入れ先出し（FIFO）順で再生しなければならない
- **FR-002**: キューが空の状態で再生リクエストが到着した場合、即座に再生を開始しなければならない
- **FR-003**: 各音声ファイルの再生は個別のオーディオ出力ストリームを使用し、再生完了後にストリームを解放しなければならない
- **FR-004**: 空文字のファイルパスはキューに追加してはならない
- **FR-005**: 存在しないファイルパスがキューに入った場合、エラーをログに記録し、次のキュー項目の再生に進まなければならない
- **FR-006**: キューへの追加はイベント処理をブロックしてはならない（非同期で行う）

### Key Entities

- **再生キュー**: 音声ファイルパスのFIFOキュー。パイプラインからの再生リクエストを受け付け、専用の再生スレッドが順番に処理する
- **再生スレッド**: キューから音声ファイルパスを取り出し、1件ずつ順番に再生する常駐プロセス

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: ALARM音の再生中にRESETイベントが発生した場合、2つの音声が重なって同時に聞こえることがない
- **SC-002**: キューが空の状態での再生開始遅延が体感できないレベルである（100ミリ秒未満）
- **SC-003**: 24時間以上の連続稼働後も、音声再生が正常に機能する
- **SC-004**: 3件以上のイベントが1秒以内に連続発生しても、すべての音声が順番に再生される

## Assumptions

- 音声ファイルの再生時間は通常1〜10秒程度であり、キューが極端に長くなることは想定しない
- オーディオ出力デバイスはシステムに1つのみ存在する
- 既存のイベントパイプライン（ALARM/RESETの発火タイミング）は変更しない
- 既存の音声ファイル選択ロジック（震度別ファイルマッピング等）は変更しない
